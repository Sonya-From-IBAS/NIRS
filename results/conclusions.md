# Исследование эффективности методов выявления вредоносного SQL кода на основе больших языковых моделей

---

## Аннотация

Работа посвящена сравнению разных подходов к обнаружению вредоносного SQL — от простых регулярных выражений до больших языковых моделей. Мы намеренно взяли задачу шире, чем классическое детектирование SQL-инъекций: помимо них рассматривались обфусцированные запросы, эскалация привилегий и скрытая эксфильтрация данных — случаи, которые в реальных атаках встречаются не реже, но исследованы значительно хуже.

Главный вопрос: могут ли LLM выступить альтернативой обученным ML-моделям там, где нет размеченных данных? Ответ оказался неоднозначным, и это делает результаты интереснее, чем если бы один подход просто победил другой.

---

## 1. Постановка задачи

Вредоносный SQL — понятие шире, чем SQL-инъекции. В работе выделено пять классов:

| Класс | Что это |
|---|---|
| legit | Обычный рабочий SQL без злого умысла |
| sqli | Классические инъекции: `' OR 1=1`, UNION SELECT, time-based |
| obfuscated | Те же атаки, но закодированные: `0x53454c454354`, CHAR(83), `/**/` вместо пробела |
| privilege | Запросы для захвата прав: GRANT ALL, xp_cmdshell, CREATE USER |
| exfiltration | Утечка данных: INTO OUTFILE, UTL_HTTP, SELECT * без WHERE на чувствительных таблицах |

Разделение obfuscated и sqli в отдельные классы — принципиальное решение. Обфусцированный запрос технически является инъекцией, но с точки зрения детектирования это другая задача: нужно распознать не паттерн атаки, а сам факт сокрытия.

---

## 2. Как строился датасет

За основу взят публичный датасет с Kaggle (около 31 тысячи примеров). Сразу обнаружилась проблема: классы privilege и exfiltration в нём практически отсутствуют. Дополнили двумя способами — сгенерировали примеры через Llama 3.3 70B и собрали реальные пейлоады из репозиториев PayloadsAllTheThings и Advanced-SQL-Injection-Cheatsheet.

Итоговое распределение неизбежно несбалансированное — это отражает реальность:

| Класс | Примеров |
|---|---|
| legit | 19 657 (62%) |
| sqli | 9 480 (30%) |
| obfuscated | 1 910 (6%) |
| privilege | 303 (1%) |
| exfiltration | 276 (1%) |

Отдельный вопрос — как размечать obfuscated. Первый вариант (любое совпадение с hex или CHAR()) давал слишком много ложных срабатываний. В итоге ввели систему баллов: пример попадает в класс obfuscated только при наличии двух и более независимых признаков кодирования. Это снизило класс с 2730 до 1910 примеров, зато разметка стала значительно чище.

Перед подачей в модели все запросы проходят нормализацию: hex-строки декодируются в ASCII, CHAR()-последовательности раскрываются, лишние комментарии убираются. Это особенно помогает при работе с обфусцированными примерами — модель видит то, что скрывает атакующий.

---

## 3. Методы детектирования

Реализовано шесть детекторов, намеренно разной природы.

**Регулярные выражения** — самый простой baseline. Четыре группы паттернов, каждая для своего класса угроз. Никакого обучения, работает мгновенно.

**ML-классификаторы** — три варианта на основе TF-IDF с символьными n-граммами (от 2 до 4 символов). Символьные n-граммы выбраны не случайно: они хорошо улавливают синтаксические особенности SQL вне зависимости от конкретных ключевых слов. Тестировали Logistic Regression, LinearSVC и Random Forest — все с `class_weight=balanced`, чтобы редкие классы не терялись.

**LLM** — Llama 3.3 70B через Groq API в двух режимах: zero-shot (только описание классов) и few-shot (плюс 14 примеров в промпте с упором на obfuscated). Чтобы не упереться в лимиты API, запросы отправлялись батчами по 10 штук.

ML-модели оценивались на полном тесте (6 326 примеров) с 5-кратной кросс-валидацией. LLM — на стратифицированной выборке из 416 примеров, иначе оценка заняла бы слишком много времени и ресурсов.

## 4. Результаты

### Кросс-валидация ML

| Метод | F1-macro |
|---|---|
| Regex | 0.706 ± 0.014 |
| TF-IDF + LogReg | 0.963 ± 0.004 |
| **TF-IDF + LinearSVC** | **0.976 ± 0.009** |
| TF-IDF + RandomForest | 0.956 ± 0.009 |

Разброс между фолдами маленький — это хороший знак. Модели не переобучились под конкретное разбиение.

## 5. Что это всё значит

**Размер модели решает больше, чем стратегия промпта.**
Мы сначала тестировали Llama 3.1 8B — результаты были удручающими (F1-macro около 0.45). На классе obfuscated модель угадывала примерно каждый двадцатый запрос (F1 = 0.05). Переход на 70B параметров поднял общий F1 до 0.70, а obfuscated — до 0.76. Это принципиальная разница: меньшая модель не просто хуже, она принципиально не справляется с задачей, которая требует понимания структуры кодирования.

**Few-shot неожиданно проиграл zero-shot.**
Интуитивно кажется, что примеры в промпте должны помогать. Но на практике few-shot (F1 = 0.62) оказался хуже zero-shot (F1 = 0.70). Виновник — класс obfuscated: при few-shot его F1 упал до 0.16, тогда как при zero-shot держался на уровне 0.76. Наиболее вероятное объяснение — при батч-классификации 10 запросов за раз 14 примеров в промпте создают слишком много контекста, и модель начинает ориентироваться на поверхностные паттерны из примеров, а не на суть. При этом на классе privilege few-shot показал лучший результат среди всех методов (F1 = 0.97) — то есть примеры помогли именно там, где нужно семантическое понимание GRANT/xp_cmdshell, а не синтаксический анализ.

**ML и LLM хороши в разном.**
ML-модели лидируют там, где важен точный синтаксический анализ: обфускация, инъекции. LLM без какого-либо обучения хорошо справляется с эскалацией привилегий (F1 = 0.83 у zero-shot против 0.88 у лучшего ML) — потому что понимает смысл GRANT ALL или xp_cmdshell из предобучения. Это намекает на гибридный подход: ML для синтаксических атак, LLM для семантических.

**Exfiltration — открытая проблема.**
Хуже всего дела с классом exfiltration. Regex угадывает только каждый шестой запрос (F1 = 0.17), LLM zero-shot — каждый второй (F1 = 0.50). Причина простая: запрос `SELECT * FROM users` снаружи ничем не отличается от легитимного, если не знать контекст — кто запрашивает, из какой таблицы, насколько широкая выборка нормальна для этого приложения. Без схемы базы данных и профиля типичного трафика ни один из рассмотренных методов не может решить эту задачу по-настоящему.

---

## 6. Выводы

Если нужен детектор прямо сейчас — **TF-IDF + LinearSVC**. F1-macro = 0.957, подтверждённый кросс-валидацией (0.976 ± 0.009), задержка 0.10 мс, никаких API и внешних зависимостей. Обучается на ноутбуке за несколько секунд.

Если данных для обучения нет — **LLM zero-shot на модели 70B+**. F1-macro = 0.702 — это уже не игрушка, а рабочий инструмент, который превосходит Regex по большинству классов и не требует разметки. Разрыв с ML составляет 1.36x — в начале работы, когда тестировали 8B-модель, этот разрыв был 2.2x. С моделями уровня GPT-4 или Claude разрыв вероятно сократится ещё.

Самый неожиданный результат — нормализация SQL как предобработка. Декодирование hex и CHAR() перед подачей в TF-IDF подняло F1 класса obfuscated у Random Forest до 1.00. Это значит, что обфускация — это по сути не другая атака, а та же инъекция в маске. Раскройте маску — и задача сводится к уже решённой.
