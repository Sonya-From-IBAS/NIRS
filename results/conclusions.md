# Исследование эффективности методов выявления вредоносного SQL кода на основе больших языковых моделей

---

## Аннотация

В данной работе проводится сравнение нескольких подходов к автоматическому обнаружению вредоносных SQL-запросов. Задача намеренно сформулирована шире, чем стандартное детектирование SQL-инъекций: рассматриваются также обфусцированные запросы, эскалация привилегий и скрытая эксфильтрация данных. Эти классы угроз встречаются в реальных атаках не реже инъекций, однако изучены значительно хуже.

Основной исследовательский вопрос — насколько большие языковые модели способны заменить классические ML-методы в условиях отсутствия размеченных обучающих данных. Результаты получились неоднозначными, что, пожалуй, делает работу интереснее, чем если бы один подход однозначно победил другой.

---

## 1. Постановка задачи

Вредоносный SQL шире, чем SQL-инъекции. В работе рассматривается пять классов запросов:

| Класс | Описание |
|---|---|
| legit | Обычный рабочий SQL |
| sqli | Классические инъекции: `' OR 1=1`, UNION SELECT, time-based |
| obfuscated | Закодированные атаки: `0x53454c454354`, CHAR(83), `/**/` вместо пробела |
| privilege | Захват прав: GRANT ALL, xp_cmdshell, CREATE USER |
| exfiltration | Утечка данных: INTO OUTFILE, UTL_HTTP, широкие SELECT без фильтрации |

Выделение obfuscated в отдельный класс от sqli — намеренное решение. С точки зрения детектирования это разные задачи: в первом случае нужно распознать факт сокрытия, во втором — саму атаку.

---

## 2. Датасет

За основу взят публичный датасет с Kaggle (~31 тысяча примеров). Главная проблема — классы privilege и exfiltration в нём почти не представлены. Для их пополнения использовались два подхода: генерация примеров через языковую модель (с разбивкой по СУБД — MySQL, MSSQL, PostgreSQL, Oracle — и двум уровням сложности запросов) и сбор реальных пейлоадов из открытых репозиториев PayloadsAllTheThings и Advanced-SQL-Injection-Cheatsheet.

Итоговый датасет (32 530 примеров):

| Класс | Примеров |
|---|---|
| legit | 19 836 (61%) |
| sqli | 9 480 (29%) |
| obfuscated | 1 910 (6%) |
| privilege | 682 (2%) |
| exfiltration | 622 (2%) |

Разметка класса obfuscated потребовала отдельного внимания. Простое правило «есть hex или CHAR() — обфускация» давало слишком много ложных срабатываний. В итоге была введена балльная система: пример относится к obfuscated только при наличии двух и более независимых признаков кодирования. Это уменьшило класс с 2730 до 1910 примеров, но сделало разметку заметно точнее.

Разбивка train/test стратифицирована по источнику данных, а не только по классу, — чтобы синтетические примеры из каждой СУБД попали и в обучение, и в тест. Без этого есть риск, что модель просто запомнит стиль генерации, а не научится детектировать реальные угрозы.

Перед подачей в модели все запросы нормализуются: hex декодируется в ASCII, CHAR()-последовательности раскрываются, inline-комментарии убираются.

---

## 3. Методы

Реализовано восемь детекторов.

**Регулярные выражения** — простейший baseline без какого-либо обучения. Четыре группы паттернов, по одной на класс угроз.

**ML-классификаторы** — TF-IDF с символьными n-граммами (2–4 символа) в связке с тремя алгоритмами: Logistic Regression, LinearSVC и Random Forest. Символьные n-граммы хорошо улавливают синтаксические особенности SQL вне зависимости от конкретных ключевых слов. Все модели обучались с `class_weight=balanced`.

**LLM-детекторы** — четыре модели в режиме few-shot (14 примеров в промпте):

| Модель | Провайдер | Режим |
|---|---|---|
| Llama 3.3 70B | Groq | zero-shot и few-shot |
| GLM-5.1 | synthetic.new | few-shot |
| gpt-oss-120b | synthetic.new | few-shot |

GLM-5.1 и gpt-oss-120b — thinking-модели: перед ответом генерируют развёрнутую цепочку рассуждений. Это влияет на скорость — 3–5 секунд на запрос против 70 мс у Llama.

ML-модели оценивались на полном тесте (6 506 примеров) с 5-кратной кросс-валидацией. LLM тестировались на стратифицированной выборке из 150–500 примеров из-за ограничений API.

## 4. Результаты

### Кросс-валидация ML

| Метод | F1-macro |
|---|---|
| Regex | 0.635 ± 0.010 |
| TF-IDF + LogReg | 0.952 ± 0.008 |
| **TF-IDF + LinearSVC** | **0.964 ± 0.008** |
| TF-IDF + RandomForest | 0.948 ± 0.008 |

Стандартное отклонение во всех случаях не превышает 0.01, что говорит об устойчивости результатов.

## 5. Обсуждение результатов

Среди LLM лучший результат показала gpt-oss-120b (F1 = 0.749), обогнав Llama 3.3 70B (0.702) и GLM-5.1 (0.708). Особенно заметна разница на классе obfuscated: 0.94 против 0.76 и 0.72 у конкурентов. По всей видимости, режим рассуждения здесь действительно помогает — модель разбирает hex-строку или CHAR()-последовательность пошагово, что сложнее сделать без явного thinking-механизма.

Вместе с тем gpt-oss-120b неожиданно слабо справляется с классом privilege (F1 = 0.57), тогда как Llama 3.3 70B в few-shot режиме показала здесь 0.97 — лучший результат среди всех методов. Это трудно объяснить однозначно: возможно, дело в том, как примеры в промпте взаимодействуют с внутренними знаниями конкретной модели о SQL-безопасности.

Отдельно стоит отметить, что у Llama few-shot оказался хуже zero-shot в целом (0.62 против 0.70). Виновник — класс obfuscated, где few-shot упал до 0.16. Судя по всему, при батч-классификации десяти запросов за раз большой промпт с примерами мешает, а не помогает: модель начинает подстраиваться под поверхностные паттерны из примеров, теряя гибкость.

ML-методы по-прежнему сохраняют значительное преимущество над LLM — разрыв составляет около 1.28x по F1-macro. Нормализация SQL перед подачей в TF-IDF оказалась неожиданно эффективной: после декодирования hex и CHAR() Random Forest достиг F1 = 1.00 на obfuscated. По сути, обфусцированный запрос — это обычная инъекция с наложенным кодированием; убрать кодирование — и задача становится стандартной.

Класс exfiltration остаётся наиболее сложным для всех подходов. Regex справляется лишь с каждым четвёртым примером (F1 = 0.25), LLM — примерно с каждым вторым (0.50–0.63). Это объяснимо: запрос `SELECT * FROM users` снаружи неотличим от легитимного без знания контекста — какие таблицы считаются чувствительными, какой объём выборки нормален для данного приложения.

---

## 6. Выводы

Для применения в системах защиты оптимальным выбором остаётся **TF-IDF + LinearSVC**: F1-macro = 0.961 при задержке 0.10 мс, результат стабилен по кросс-валидации (0.964 ± 0.008), не требует внешних зависимостей.

В сценарии, когда размеченных данных нет и обучение невозможно, наиболее подходящим вариантом выглядит **gpt-oss-120b** в режиме few-shot: F1 = 0.749 при разрыве с лучшим ML около 1.28x. Основное ограничение — задержка около трёх секунд на запрос и зависимость от внешнего API.
