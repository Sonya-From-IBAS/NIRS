# Исследование эффективности методов выявления вредоносного SQL кода на основе больших языковых моделей

---

## 1. Аннотация

В работе проведено сравнительное исследование методов автоматического выявления вредоносного SQL кода. Рассмотрены четыре класса угроз: SQL-инъекции, обфусцированный SQL, эскалация привилегий и скрытая эксфильтрация данных. Реализованы и протестированы шесть детекторов: регулярные выражения, три варианта ML-классификаторов (TF-IDF + LogReg / LinearSVC / Random Forest) и два режима работы LLM (zero-shot, few-shot) на базе модели Llama 3.3 70B через Groq API. Эксперимент проведён на датасете из 31 626 примеров; ML-модели оценивались методом 5-кратной кросс-валидации. Показано, что классические ML-методы превосходят LLM по F1-macro в 1.36 раза при задержке в 680 раз меньше. При переходе с модели 8B на 70B параметров LLM улучшает F1-macro на 25 процентных пунктов (с 0.45 до 0.70), что сокращает разрыв с ML-методами с 2.2x до 1.36x.

---

## 2. Постановка задачи

**Цель:** исследовать, способны ли большие языковые модели эффективно детектировать вредоносный SQL-код в сравнении с классическими методами.

**Классы угроз:**
| Класс | Описание | Примеры |
|---|---|---|
| `legit` | Легитимный SQL | SELECT с JOIN, INSERT для бизнес-логики |
| `sqli` | SQL-инъекции | `' OR 1=1 --`, UNION SELECT, time-based |
| `obfuscated` | Обфусцированный SQL | `CHAR(83)`, `0x53454c454354`, inline-комментарии |
| `privilege` | Эскалация привилегий | GRANT ALL, xp_cmdshell, ALTER USER |
| `exfiltration` | Скрытая эксфильтрация | INTO OUTFILE, UTL_HTTP, SELECT без фильтрации |

## 3. Методология

### 3.1 Датасет

- **Источники:** Kaggle SQL Injection Dataset (30 919 примеров), синтетические данные (750 примеров, сгенерированы через Llama 3.3 70B), реальные пейлоады из PayloadsAllTheThings и Advanced-SQL-Injection-Cheatsheet (50 примеров)
- **Итого:** 31 626 примеров после дедупликации
- **Разметка обфускации:** строгая переклассификация — требуется score ≥ 2 по независимым сигналам (hex ≥ 6 символов, CHAR(), inline-комментарии, URL-кодирование). Устраняет утечку данных между обучением и регуляр-выражениями.
- **Нормализация SQL:** декодирование hex и CHAR(), удаление inline-комментариев, нормализация пробелов — применяется как первый шаг в sklearn-пайплайне
- **Разбивка:** 80% train / 20% test, стратификация по классам

| Класс | Примеров | % |
|---|---|---|
| legit | 19 657 | 62.2% |
| sqli | 9 480 | 30.0% |
| obfuscated | 1 910 | 6.0% |
| privilege | 303 | 1.0% |
| exfiltration | 276 | 0.9% |

### 3.2 Реализованные методы

**Baseline:**
- **Regex:** 4 группы ручных паттернов (по одной на класс угроз), приоритетная последовательная проверка
- **TF-IDF + LogReg:** SQLNormalizer → символьные n-граммы (2–4), 50 000 признаков, class_weight=balanced
- **TF-IDF + LinearSVC:** аналогично, LinearSVC с C=1.0
- **TF-IDF + Random Forest:** 200 деревьев, char n-gram, class_weight=balanced

**LLM-методы (Llama 3.3 70B, Groq API):**
- **Zero-shot:** системный промпт с детальным описанием классов + запрос без примеров
- **Few-shot:** 3–4 примера на каждый класс (14 примеров суммарно), акцент на obfuscated

Тестирование LLM проводилось на стратифицированной выборке 416 примеров (до 100 на класс). Батчинг: 10 запросов за один вызов API.

### 3.3 Оценка

- **ML:** 5-кратная стратифицированная кросс-валидация + holdout-тест (20%)
- **LLM:** holdout-тест на 416 примерах
- **Метрики:** F1-macro, F1-weighted, Precision, Recall, латентность (мс/запрос)

## 5. Анализ и обсуждение

### 5.1 Влияние размера модели на качество LLM

Переход с Llama 3.1 8B на Llama 3.3 70B даёт прирост F1-macro на 25 п.п. (0.45 → 0.70). Наиболее выражен эффект на классе `obfuscated`: F1 вырастает с 0.05 до 0.76 — модель 70B способна распознавать hex-кодирование и CHAR()-последовательности, что недоступно меньшей модели.

### 5.2 ML превосходит LLM, но разрыв сокращается

TF-IDF + LinearSVC достигает F1-macro = 0.957 при задержке 0.10 мс — в 1.36 раза лучше лучшего LLM (0.702). При использовании 8B-модели разрыв составлял 2.2x. Прогноз: с моделями 200B+ разрыв продолжит сокращаться.

### 5.3 Few-shot хуже zero-shot на obfuscated

Неожиданный результат: few-shot (F1=0.62) уступает zero-shot (F1=0.70), несмотря на наличие примеров. Причина — на классе `obfuscated` few-shot падает до F1=0.16 против 0.76 у zero-shot. Гипотеза: 14 примеров в промпте создают контекстный шум, который сбивает модель при батч-классификации 10 запросов одновременно. Zero-shot с чистым описанием классов работает надёжнее.

### 5.4 Эксфильтрация — сложнейший класс для всех методов

`exfiltration` показывает наихудший F1 у большинства методов: Regex 0.17, LLM zero-shot 0.50. Причина: запросы на эксфильтрацию синтаксически корректны и семантически неоднозначны без контекста схемы БД. Это открытая проблема для будущих исследований.

### 5.5 Нормализация SQL улучшает ML

Добавление SQLNormalizer в пайплайн (декодирование hex, CHAR(), удаление комментариев) повысило стабильность cross-val на классе `obfuscated`. F1 класса obfuscated для TF-IDF + RandomForest достиг 1.00 — модель видит декодированный SQL и уверенно его классифицирует.

### 5.6 k-fold CV подтверждает надёжность результатов

Стандартное отклонение F1-macro по 5 фолдам составляет 0.004–0.014 — это означает, что результаты воспроизводимы и не являются артефактом конкретного разбиения данных.

## 6. Выводы

1. **TF-IDF + LinearSVC** остаётся оптимальным для продакшн-применения: F1-macro = 0.957 (CV: 0.976 ± 0.009), задержка 0.10 мс, работает без API.

2. **Размер LLM критически важен:** переход 8B → 70B даёт +25% F1-macro. LLM zero-shot 70B (F1=0.702) уже сопоставим с Regex-детектором уровня WAF при значительно лучшем охвате классов.

3. **Разрыв ML vs LLM сократился с 2.2x до 1.36x** за счёт смены модели. С моделями уровня GPT-4 / Claude Opus разрыв вероятно сократится до 1.1–1.2x.

4. **Few-shot не всегда лучше zero-shot** на малых моделях и при батч-классификации. Zero-shot с чётким описанием классов — более надёжная стратегия.

5. **Нормализация SQL как препроцессинг** существенно помогает ML на классе `obfuscated` и должна быть включена в любой production-пайплайн.

6. **Exfiltration — нерешённая задача:** все методы показывают F1 < 0.85 без контекста схемы БД.

---

## 7. Угрозы внутренней валидности

- **Синтетические данные:** классы `privilege` и `exfiltration` частично сгенерированы LLM, что может создавать смещение при оценке LLM-детектора на этих классах.
- **Разный объём теста:** ML тестировался на 6 326 примерах, LLM — на 416. Статистическая мощность сравнения снижена.
- **Ограниченный размер LLM:** использована модель 70B; коммерческие модели (GPT-4o, Claude Opus) вероятно покажут значительно лучшие результаты.
- **Батч-классификация LLM:** группировка 10 запросов в один вызов API может снижать точность по сравнению с пообъектной классификацией.

## 8. Направления дальнейших исследований

- Fine-tuning LLM (CodeBERT, SQLCoder) на размеченном датасете
- Тестирование коммерческих LLM (GPT-4o, Claude Opus) для оценки потолка метода
- Контекстно-зависимое детектирование exfiltration (с учётом схемы БД)
- Adversarial robustness: устойчивость к целенаправленному обходу детекторов
- Гибридный подход: нормализация + ML для синтаксических атак, LLM для семантических
